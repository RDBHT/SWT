REF-E1 — Refactoring-Katalog

Einsendeaufgabe REF-E1 (Softwaretechnik)

Auf Basis des Refactoring-Katalogs von Martin Fowler: Eindrücke zum Katalog, die Refactoring-Unterstützung der eigenen IDE, ein eigenes Vorher/Nachher-Beispiel und ein Refactoring mit generativer KI. Die Code-Beispiele stammen aus der Domäne des IT-Service-Monitors.

Teil 1 — Eindrücke zum Katalog

Beeindruckt haben mich zwei Refactorings: Replace Conditional with Polymorphism — es ersetzt wachsende Typ-Fallunterscheidungen durch eine Klassenhierarchie und ändert damit die Struktur, nicht nur die Optik. Und Split Phase — es trennt einen Block, der zwei Dinge tut, in zwei Phasen mit Zwischendatenstruktur; es benennt ein Problem, das ich oft gespürt, aber nie gefasst hatte.

Überflüssig finde ich Slide Statements: zusammengehörige Zeilen zusammenzurücken ist für mich Schreibstil, kein eigener Katalogeintrag. Nicht sofort verstanden habe ich Replace Subclass with Delegate — wann sich Delegation gegenüber Vererbung wirklich lohnt, erschloss sich mir erst am zweiten Beispiel.

Teil 2 — IDE: Visual Studio Code

Ich arbeite mit VS Code; die Refactorings für JavaScript/TypeScript kommen aus dem eingebauten TypeScript-Language-Service (Strg+.).

VS Code deckt die kleinen, lokalen Refactorings zuverlässig ab. Die großen, strukturellen (Replace Conditional with Polymorphism, Extract Class) sind nicht automatisiert — sie entstehen aus mehreren kleinen Schritten plus Handarbeit.

Teil 3 — Eigenes Beispiel: Replace Conditional with Polymorphism

Im IT-Service-Monitor entscheidet eine Funktion per switch über den Check-Typ — jeder neue Typ vergrößert den switch. Mit VS Code (Extract to function, Move to new file, Rename) wird daraus eine Klassenhierarchie.

Vorher

function runCheck(check) {
  switch (check.type) {
    case 'http': {
      const res = httpGet(check.url, check.timeout);
      return { status: res.code === check.expected ? 'OK' : 'DOWN', latency: res.ms };
    }
    case 'tcp': {
      const probe = tcpProbe(check.host, check.port, check.timeout);
      return { status: probe.open ? 'OK' : 'DOWN', latency: probe.ms };
    }
    case 'dns': {
      const rec = dnsResolve(check.host, check.recordType);
      return { status: rec.found ? 'OK' : 'DOWN', latency: rec.ms };
    }
    default:
      throw new Error('Unbekannter Check-Typ: ' + check.type);
  }
}

Nachher — abstrakte Oberklasse plus eine Klasse je Typ:

class Check {
  constructor(config) { this.config = config; }
  execute() { throw new Error('execute() muss überschrieben werden'); }
}

class HttpCheck extends Check {
  execute() {
    const res = httpGet(this.config.url, this.config.timeout);
    return { status: res.code === this.config.expected ? 'OK' : 'DOWN', latency: res.ms };
  }
}

class TcpCheck extends Check {
  execute() {
    const probe = tcpProbe(this.config.host, this.config.port, this.config.timeout);
    return { status: probe.open ? 'OK' : 'DOWN', latency: probe.ms };
  }
}

class DnsCheck extends Check {
  execute() {
    const rec = dnsResolve(this.config.host, this.config.recordType);
    return { status: rec.found ? 'OK' : 'DOWN', latency: rec.ms };
  }
}

Die Typunterscheidung verschwindet nicht ganz — sie wandert an genau eine Stelle, eine kleine Factory. Ein neuer Check-Typ ist danach eine neue Klasse, der bestehende Code bleibt unangetastet (Open-Closed-Prinzip).

Teil 4 — Refactoring mit generativer KI

Eine überladene Funktion evaluateService() sollte die KI (Claude) per Extract Function und Decompose Conditional refactoren — verhaltensgleich.

Hat es sofort geklappt? Nein. Der erste Versuch extrahierte zwar sinnvoll, modernisierte aber ungefragt mit: for-Schleifen wurden zu reduce/filter, eine Variable umbenannt. Das ist lesbarer, aber nicht mehr „nur extrahiert" — und barg ein Verhaltensrisiko. Erst der Nachfass-Prompt („nur extrahieren, Schleifen und Logik unverändert") brachte eine konservative, verhaltensgleiche Fassung.

Erkenntnis: Die KI erkennt Extraktionsgrenzen gut, neigt aber dazu, mehr zu „verbessern" als verlangt. Refactoring heißt Struktur ändern, Verhalten nicht — diese Disziplin muss man der KI ausdrücklich auferlegen.
