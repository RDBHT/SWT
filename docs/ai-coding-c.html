Verteiltheit — drei Prozesse

Teil C ist verteilt im Sinne der Vorlesung: nicht ein Prozess mit Frontend/Controller/Backend, sondern drei eigenständige Prozesse (JVMs), die über HTTP kommunizieren. Der agent prüft und sendet, der collector verarbeitet und zeigt an.

flowchart LR
    subgraph P1["Prozess 1 · mock-service"]
      mock["GET /health"]
    end
    subgraph P2["Prozess 2 · agent"]
      sched["scheduler + checker"]
      client["IngestClient"]
    end
    subgraph P3["Prozess 3 · collector"]
      ingest["POST /ingest
store + alerting"]
      dash["GET / Dashboard"]
    end
    browser(["Browser"])
    sched -- "HTTP-Check" --> mock
    client -- "HTTP POST /ingest" --> ingest
    browser -- "GET /" --> dash

Die Naht im Code ist das ResultSink-Interface: der Scheduler kennt weder store noch alerting; im verteilten Betrieb ist der Sink der IngestClient (HTTP-POST). Der Agent kann auf einem anderen Rechner laufen — nur die Collector-URL ändert sich.

Starten (drei Terminals):

mvn -q -DskipTests install
mvn -q -pl mock-service exec:java   # Prozess 1 (:8081)
mvn -q -pl collector   exec:java    # Prozess 3 (:8080)
mvn -q -pl agent       exec:java    # Prozess 2
