ARC-E2 — Architecture Communication Canvas abgegeben

Ein Architecture Communication Canvas (nach canvas.arc42.org) für einen kleinen Teil der ARC-E1-Architektur: die Alerting-Pipeline des IT-Service-Monitors — bekannt aus TST-E1 (Mocking-Tests) und MET-E1 (Komplexitäts-Hotspot).

Canvas: Alerting-Pipeline

Value Proposition

Alarmiert Betreiber zuverlässig und ohne Fehlalarm-Flut: erst nach Überschreiten des konfigurierten DOWN-Fensters wird genau ein Alert ausgelöst — kein Spam bei flatternden Diensten, keine Doppel-Alerts.

Key Stakeholder

  • Operator/Admin — schnelle, verlässliche Alarme
  • Entwicklerteam — mockbare Schnittstellen
  • Kursbetreuung — bewertet Architektur- und Testqualität

Core Functions

  • Statusereignisse konsumieren (tick je Ziel)
  • DOWN-Strecken erkennen und entprellen (alertAfterMs)
  • Genau einen Alert an den AlertSink feuern
  • Erholung erkennen, Alarmzustand zurücksetzen

Quality Requirements

  • Zuverlässigkeit: kein verlorener/doppelter Alert
  • Testbarkeit: ohne echte Uhr/Netz/Mail (Mockito)
  • Erweiterbarkeit: neue Kanäle ohne Pipeline-Änderung
  • Wartbarkeit: Hotspot unter Metrik-Beobachtung (tick: McCabe 5, Cognitive Complexity 6)

Business Context

  • Input: UP/DEGRADED/DOWN vom StatusEvaluator, Zeit via injizierter Clock
  • Output: Alerts an MailSink / WebhookSink → Operator
  • Nachbarn: scheduler (Takt), store (unabhängige Persistenz)

Core Decisions — good or bad

  • ✅ Entprellung über Zeitfenster statt Zähler — robust bei variablen Intervallen
  • ✅ AlertSink als Interface (Strategy) — Kanäle austauschbar
  • ✅ Injizierte Clock — deterministische Zeitreise-Tests
  • ⚠️ Alarmzustand nur im Speicher — geht bei Neustart verloren (KISS, bewusst akzeptiert)

Components / Modules

AlertingService (Zustandsmaschine) · StatusEvaluator (Ereignisquelle) · AlertSink-Interface + Adapter · injizierte Clock

Core Technologies

Java 17 · Maven · JUnit 5 + Mockito 5 · PMD/Checkstyle-Gate im CI (MET-E1)

Risks & Missing Information

  • Zustandsverlust bei Agent-Neustart — Persistenz-Entscheidung offen
  • Eskalation/Sammel-Alert bei vielen gleichzeitigen Ausfällen nicht spezifiziert
  • Zustellgarantie: Retry-Strategie bei fehlschlagendem Mail-/Webhook-Versand ungeklärt
