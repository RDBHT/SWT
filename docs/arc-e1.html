ARC-E1 — Finale Architektur in Bearbeitung

Konkreter Architekturentwurf für das durchgängige Start-Up-Projekt IT-Service-Monitor (HTTP-/TCP-/DNS-Health-Checks, Verlauf, Alerting — siehe VOR-E1, AI-Coding, TST-E1, MET-E1). Zentrales Element: das Komponentendiagramm mit vier Ebenen. Vollständige Erläuterung im README.

Verteilungsschnitt: Check-Agent (scheduler + checker + alerting) und dashboard-ui sind getrennt deploybar; einzige Koppelstelle ist die Store-API (Dashboard read-only).

Erläuterung in Kürze (Lerneinheit ARC)

  • Prinzipien: Separation of Concerns (ein Modul, eine Verantwortung), Information Hiding (schmale APIs: Check, Store, AlertSink), Strategy für Check-Typen und Alert-Kanäle, Dependency Injection (Mockbarkeit, vgl. TST-E1), KISS/YAGNI (kein Broker, kein Microservice-Zoo).
  • Ebenen: vier — Präsentation, Anwendung, Domäne, Infrastruktur; Abhängigkeiten nur nach unten, Domäne framework-frei.
  • Architekturstil: komponentenbasiert/geschichtet als Grundgerüst, Pipe-and-Filter im Messpfad (scheduler → checker → StatusEvaluator → store/alerting), Client-Server zwischen Dashboard und Store.
  • Frameworks: Java 17 + Maven-Multi-Module, JUnit 5 / Mockito 5, SQLite via JDBC, statisches HTML/JS-Dashboard, GitHub Actions — bewusst ohne Spring Boot zum Start.
  • Qualität & Metriken: CI-Quality-Gate mit PMD/Checkstyle (Rulesets aus MET-E1), JaCoCo, punktuell PIT; Ebenenregel und Zyklenfreiheit per ArchUnit/JDepend im Build geprüft. Fokus auf die bekannten Hotspots (alerting-Zustandslogik, Validierungscode).
