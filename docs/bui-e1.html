BUI-E1 — Buildmanagement

Erfahrung mit zwei Build-/CI-Systemen: Maven als Buildmanagementsystem und GitHub Actions als CI/CD-Orchestrator, der denselben Maven-Build automatisiert ausführt. Repository: github.com/RDBHT/SWT.

1. Gewählte Systeme und Begründung

Bewusst nicht zwei CI/CD-Tools — die Kombination Buildsystem + CI/CD gibt den größeren Lern-Kontrast: Actions baut nicht selbst, sondern ruft Maven auf.

2. Beispielprojekt — was konkret passiert

Eine Utility-Klasse plus zwei JUnit-5-Tests — es geht um das Prinzip der Pipeline, nicht um Umfang.

bui-e1/maven-demo/
├── pom.xml
└── src/
    ├── main/java/de/rdbht/swt/Anonymizer.java       die Logik
    └── test/java/de/rdbht/swt/AnonymizerTest.java   2 JUnit-5-Tests

Logik (Anonymizer.maskEmail): nimmt eine E-Mail-Adresse, prüft auf genau ein @ (sonst IllegalArgumentException), behält das erste Zeichen des lokalen Teils, ersetzt den Rest durch *** und hängt die Domain wieder an — "anna@example.de" → "a***@example.de". Eine main-Methode gibt das Ergebnis aus, damit das JAR direkt ausführbar ist.

Tests (AnonymizerTest): (1) prüft die korrekte Maskierung, (2) prüft, dass eine Eingabe ohne @ die erwartete Exception wirft.

Warum genau das? Jede Datei deckt eine Build-Phase ab und macht sie beobachtbar: Quellcode muss kompilieren, Tests müssen in der test-Phase grün sein, erst dann entsteht in der package-Phase das JAR, das sich ausführen lässt. Schlägt eine Stufe fehl, bricht der Build ab (Abschnitt 5).

3. Maven — lokaler Build

Die Umgebung war zunächst leer (nur ein Java-JRE, kein Compiler, kein Maven). Aufgesetzt ohne Root-Rechte, rein über entpackte Tarballs ins Home-Verzeichnis: 

# 1) Volles JDK 17 (Temurin) — enthaelt javac (das reine JRE nicht)
curl -L -o jdk17.tar.gz \
  "https://api.adoptium.net/v3/binary/latest/17/ga/linux/aarch64/jdk/hotspot/normal/eclipse"
tar xzf jdk17.tar.gz

# 2) Maven 3.9.9 (Binary)
curl -L -o maven.tar.gz \
  "https://archive.apache.org/dist/maven/maven-3/3.9.9/binaries/apache-maven-3.9.9-bin.tar.gz"
tar xzf maven.tar.gz

# 3) Einhaengen + pruefen
export JAVA_HOME="$PWD/jdk-17.0.19+10"
export PATH="$JAVA_HOME/bin:$PWD/apache-maven-3.9.9/bin:$PATH"
mvn -version   # Apache Maven 3.9.9 / Java 17.0.19, Eclipse Adoptium

Anschließend der Build — cd bui-e1/maven-demo && mvn -B clean package:

$ java -jar target/maven-demo-1.0.0.jar
Input : anna@example.de
Masked: a***@example.de

4. Beobachtung: Dependency-Cache

Maven cached Plugins/Bibliotheken in ~/.m2. Genau das bildet die CI über cache: maven nach.

5. Test-Gate: roter Test stoppt den Build

[ERROR] AnonymizerTest.masksLocalPart:14 expected: <XXX@example.de> but was: <a***@example.de>
[INFO] BUILD FAILURE
[ERROR] ...maven-surefire-plugin:3.2.5:test ...: There are test failures.

Exit-Code ≠ 0, die package-Phase wird nicht erreicht — kein JAR. Genau dieser Exit-Code ist die Brücke zur CI: in der Pipeline läuft derselbe Befehl als Schritt run: mvn -B clean package. GitHub Actions wertet den Exit-Code aus — 0 = Schritt grün, ≠ 0 = Schritt schlägt fehl, folgende Schritte werden übersprungen, Job und Run werden rot, und das Badge oben wird rot. Ein fehlerhafter Commit fällt so automatisch auf, bevor er als funktionierend gilt.

6. GitHub Actions — CI/CD

Workflow .github/workflows/build.yml: bei jedem Push Checkout → JDK 17 + Maven-Cache → mvn -B clean package → JAR & Testberichte als Artefakt.

  • Live-Status: Badge oben auf dieser Seite
  • Konkrete Run-Logs: Actions-Tab des Repos (erster Run folgt nach dem Push)

8. Fazit

Maven liefert durch Konventionen + Lifecycle schnell einen sauberen Standardbuild; GitHub Actions automatisiert genau diesen Build reproduzierbar in der Cloud. Kernerkenntnis: CI/CD ersetzt kein Buildsystem, es automatisiert es — erst die Kombination ergibt eine vollständige Pipeline.
